## 全体のたとえ：API取得は「出前の注文」

このノートの説明は、ずっと**出前（デリバリー）**のたとえを使います。

| プログラムの言葉 | 出前で言うと |
| --- | --- |
| URL | 注文する「お店の住所＋注文内容」を書いたメモ |
| fetch | その住所まで「取りに行ってきて」という指示 |
| await | 出前が届くまで「待つ」こと |
| response | 届いたけど、まだ袋を開けていない出前 |
| .json() | 袋を開けて、中身を食べられる形にすること |

---

## 1. APIには「鍵が要るもの」と「要らないもの」がある

| タイプ | たとえ | 今回のAPI |
| --- | --- | --- |
| 鍵（APIキー）が要る | 会員証が必要なお店 | さくらのAI Engine（AI①・②） |
| 鍵が要らない | 誰でも入れる無料の公共サービス | Quotable（名言取得） |

- **Quotable** … 無料 → **APIキー不要**。住所に行って「ください」と言うだけ。
- **さくらのAI Engine** … 有料枠（月3,000回） → **APIキー必要**（`config.js` の `SAKURA_API_KEY`）。

---

## 2. URL・GET・fetch とは

- **URL** … インターネット上の「住所」。「どこに・何を頼むか」を表す文字列。
- **GET** … その住所に「これ、ちょうだい」とお願いして受け取ること。
- **fetch** … JavaScriptで「その住所まで取りに行ってきて」と命令する道具。

```
https://api.quotable.io/quotes/random?tags=wisdom
```
- `?` より前 … お店の住所（ランダムな名言をください）
- `?tags=wisdom` … 注文の条件（「wisdom」の名言で）

---

## 3. ドキュメントの「住所の書き方」のお約束

ドキュメントの `GET /quotes/random` は**住所の「後半（パス）」だけ**。
前半（ベースURL `https://api.quotable.io`）はドキュメントの最初に1回だけ書いてある。

| 部分 | たとえ |
| --- | --- |
| ベースURL（`https://api.quotable.io`） | ビルの住所（全部屋で共通だから1回だけ） |
| パス（`/quotes/random`） | その中の部屋番号 |

前半＋後半をくっつけると、実際にアクセスするフルの住所になる。

> 似た見出しに注意：**Get Random Quotes**（複数形→`/quotes/random`・推奨）と
> **Get random quote**（単数形→`/random`・非推奨）。

---

## 4. 文字を「+」でくっつける

```javascript
"おはよう" + "ございます"   →   "おはようございます"
```
住所づくりも同じ：
```javascript
const url = "https://api.quotable.io/quotes/random?tags=" + tag;
```
- `"..."` … いつも同じ固定の住所
- `tag` … 箱の名前。中身（例 `"Failure"`）に置き換わる
- `tag` は「ランダム」ではなく、**そのとき渡された値**が入るだけ。

---

## 5. URLは「文字列」でOK（HTMLじゃない）

`<script>` の中は **JavaScript（命令を書く場所）**。HTMLを作る場所ではない。
URL（住所）はもともとただの文字なので、`"..."` で囲んだ**文字列**で書くのが正解。

---

## 6. 【重要なワナ】全角と半角

コードの記号は必ず**半角（英数モード）**で打つ。
- ❌ `；`（全角）… プログラムは読めずエラー
- ✅ `;`（半角）… 正解

> `tags` の `s` 抜けのような小さなスペルミスも要注意。

---

## 7. 関数（function）＝「名前のついた機械」

```
async   function   fetchQuote   (tag)
 └④┘    └─①─┘     └──②──┘     └③┘
```
- ① `function` … ひとまとまりの作業に名前をつけた「機械」。🐟たい焼き機（材料→処理→できあがり）。
- ② `fetchQuote` … 機械の名前。`fetchQuote("Wisdom")` のように呼ぶと動く。
- ③ `(tag)` … 機械に入れる材料（引数）。🐟あんこ。中で `tag` と書くと「入れられた材料」を指す。
- ④ `async` … 「途中で“待つ”ことがある機械ですよ」の印。`await` とセット。

```javascript
async function fetchQuote(tag) {   // 入口。材料は tag。待ちあり。
   ...処理...                      // 中の作業
   return { content: ..., author: ... };  // できあがりを出す
}
```

---

## 8. `await` ＝「ここで待ってね」

`await` は「この作業が終わる（返事が来る）まで、ここで待ってから次に進んで」。
`fetch` はインターネット越しで時間がかかるので、せっかちなプログラムに**ブレーキ**をかける。

🍽 出前：`await`なし＝注文した瞬間に食べようとする（手元に何もない）／`await`あり＝届くまで待つ。

> 覚え方：`fetch` や `.json()` など、**時間がかかる作業の前には `await`**。

---

## 9. `.json()` ＝「袋を開けて、読める形にする」

```javascript
const data = await response.json();
```
- `response.json()` … 封がされた箱（response）を開けて、JavaScriptで扱える形に変換
- `await` … 変換にも少し時間がかかるので待つ
- `const data =` … 取り出した中身を `data` に入れる

🍽 「届いた出前の袋を開けて、料理をお皿に並べた」状態。

---

## 10. APIは全部JSON？ → いいえ、よくある形の1つ

- **JSON** … データを文字で整理して書く「書き方のルール」。`{ "名前": 値 }` の形。多くのWeb APIが使う。
- でも全部ではない。中身の形に合った「開け方」を選ぶ：

| 形 | 開け方 |
| --- | --- |
| JSON（今回） | `.json()` |
| ふつうの文字・HTML | `.text()` |
| 画像・ファイル | `.blob()` |

🍽 和食ならお箸（`.json()`）、洋食ならナイフとフォーク（`.text()`）。
今回のQuotableはJSONなので `.json()` でOK。

---

## 11. `[]`（配列）と `Array<...>` は同じもの

ドキュメントの `Array<{...}>` は **形の“説明書き”**。実際のデータは `[ {...} ]`（`[]`の配列）。

```javascript
[
  { "content": "名言...", "author": "Steve Jobs", "tags": ["wisdom"] }
]
```
- いちばん外側が `[ ]`（配列）、中に `{ }`（オブジェクト）。
- `Array<{...}>` ＝ 「`[ {...} ]` のことだよ」という説明。別物ではない。

---

## 12. オブジェクトから中身を取り出す（`.名前`）

`{ }` は🗄**名札のついた引き出し**。「名前のあとに `.名札`」で中身が出る。

```javascript
data[0].content   →  名言本文
data[0].author    →  作者名
```
- `data` … 配列まるごと `[ {...} ]`
- `data[0]` … 配列の**1個目**（`[0]`＝0番目＝先頭。プログラムは0から数える）
- `.content` / `.author` … その中の引き出しを開ける

> ⚠️ これは「本家Quotable」の形。**ミラー版は形が違う**ので、取り出し方も変わる（下の13参照）。

## 13. 【今回の壁】本家が使えない → ミラー版に切り替え

### 何が起きた？
ブラウザのConsoleにこのエラー：
```
GET https://api.quotable.io/quotes/random?tags=wisdom  net::ERR_CERT_DATE_INVALID
```
→ **Quotableサーバーの安全証明書（SSL証明書）の期限切れ**。ブラウザが安全のためブロック。
🏪 お店の営業許可証の期限が切れていた状態。**自分のコードのミスではない**。

> `処理に失敗しました：TypeError: Failed to fetch` は、自分で書いた `catch` が
> ちゃんと働いてエラー画面に切り替えた証拠（設計どおり動いている）。
> 一番上の `The message port closed...` はブラウザ拡張のノイズで無関係。

### 解決：Quotableのミラー版（姉妹店）に切り替え
同じ仕組みを別の場所で動かしている有志のミラーを使う（鍵不要・今動いている）。

| | 本家（使えない） | ミラー版（使える） |
| --- | --- | --- |
| 住所 | `https://api.quotable.io/quotes/random?tags=` | `https://api.quotable.kurokeita.dev/api/quotes/random?tags=` |
| タグ | 小文字 `wisdom` | **大文字始まり** `Wisdom`（小文字だと0件） |
| 返事の形 | `[ { content, author } ]`（配列） | `{ quote: { content, author: { name } } }`（quoteの箱に包む） |

### ミラー版での取り出し方（本家と違う！）
```javascript
data.quote.content       →  名言本文
data.quote.author.name   →  作者名（authorがさらに箱になっていて、その中の name）
```
※ ミラー版は配列ではなく `quote` という箱が1つなので `[0]` は不要。
※ ただし**必ず console.log で実物を見てから**確定する（設計書の約束）。

## 14. 「このAPI使えるかな？」の見分け方

ページの見た目では判断できない（プロでも見た目だけでは判断しない）。この5つで見る：

| チェック | 見るところ |
| --- | --- |
| ① 説明書があるか | ドキュメント / Swagger UI があるか |
| ② **実際に叩くとデータが返るか** | URLを叩いてみる（いちばん確実・王道） |
| ③ 鍵が要るか | 「API key」「token」の記載 |
| ④ ブラウザから使えるか | CORS許可があるか（実際に試すと分かる） |
| ⑤ 無料か | 料金の記載 |

> いちばん大事なのは ②「実際に叩いてみる」。説明書が立派でも動かないことはあるし、
> 地味なページでもちゃんと動くことがある。「動かして確かめる」が王道（＝`console.log`で確認と同じ発想）。

### 「人間向けの入口」と「プログラム向けの入口」は別
- `quotable.kurokeita.dev`（**api.なし**）… 人が読む**紹介ページ**（お店の正面入口・看板）
- `api.quotable.kurokeita.dev`（**api.あり**）… プログラムが叩く**API**（裏口・業務用の受け渡し口）
- 多くのサービスで、APIの住所は頭に **`api.`** が付く慣習。コードで使うのは `api.` 付きの方。
- Swagger UI … APIの入口一覧をその場で試せる「対話できる説明書」。これがある＝生きているAPIの良い目印。
