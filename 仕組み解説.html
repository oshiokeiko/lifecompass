仕組み解説

「人生の羅針盤」は、内側でどう動いているのか？
──気持ちを入れてから名言が出るまでを、たとえ話で1枚に。

このページの読み方

このアプリの仕組みを、いつも次の4ステップで分解して説明します。

①名札HTMLの部品に付けた「名前（id）」。どの部品か指で差せるようにする目印。
②結びつけその名札を頼りに、JavaScript（jQuery）が部品を「つかむ」。
③処理つかんだ部品を使って、実際の作業（AIやAPIの呼び出し）をする。
④何が起きる結果として、画面の表示がどう変わるか。
全体のたとえ：API＝出前（デリバリー）、APIキー＝会員証。 この2つのたとえが、ずっと出てきます。

1. 全体の流れ（このアプリの心臓部）

気持ちを入れてから名言が出るまで、AIを2回・名言APIを1回、合計3回の「出前」を順番に頼んでいます。前の出前の中身を、次の注文に使うのがポイントです。

あなたの気持ち
例）仕事で失敗して落ち込んでいる
▼
AI①：タグ選択（さくらのAI Engine）
候補から合うタグを1つ選ぶ → 例 {"tag":"Failure"}
▼
名言API：Quotable
そのタグで英語の名言を1つ取ってくる（鍵なし）
▼
AI②：翻訳＋やさしい一言（さくらのAI Engine）
英語を日本語に訳し、寄り添う一言を添える
▼
画面に表示
日本語訳・原文・作者・AIの一言

AIは「名言を作る」のではなく、気持ちを読む・翻訳するという橋渡しに徹します。名言そのものは必ずQuotableから取るので、出典の捏造が起きません。

2. 「名言を探す」を押すと何が起きる？

いちばん大事な操作を、①名札→②結びつけ→③処理→④何が起きる で分解します。

①名札（id） ボタンに id="submit-btn"、入力欄に id="feeling-input" という名札を付けてある。
②結びつけ jQuery が名札でボタンをつかみ、「クリックされたら動け」と見張りを付ける： 
$('#submit-btn').on('click', handleSubmit);
#submit-btn の # は「id を名指しする」記号。.on('click', …) は「押されたら呼んでね」の予約。
③処理 handleSubmit が動き、3つの出前を順番に頼む： 
const tag       = await chooseTag(feeling);        // AI①
const quote     = await fetchQuote(tag);           // 名言API
const translated = await translateQuote(quote, …); // AI②
await＝「届くまで待ってから次へ」。前の結果（tag→quote）を次に手渡している。
④何が起きる 結果を画面に流し込み、結果画面に切り替わる。途中で失敗したら catch が受け止めてエラー画面へ（白画面で固まらない）。

3. AI①「タグ選択」の中身

気持ちを読み、候補の中から合うタグを1つ選ばせる部分（chooseTag）。ここで初めて会員証（APIキー）を使います。

①宛先を決めるさくらのAI Engine の住所（出前のお店）を用意：https://api.ai.sakura.ad.jp/v1/chat/completions
②お願い文（プロンプト）を書くsystem＝AIのルール（「候補から1つだけ、JSONで返して」）、user＝あなたの気持ち。候補タグも一緒に渡す。
③送る（鍵を見せる）fetch で POST。ヘッダに Authorization: "Bearer " + キー を付けて「私は会員です」と提示。
④受け取って取り出す返事を .json() でほどき、data.choices[0].message.content からAIの文を取り出す。
防御的パース：AIが余計な前置きを付けても大丈夫なように、文章から最初の { ～ 最後の } だけを切り出して JSON.parse する。 さらに、選ばれたタグが候補に本当に入っているかを最後に確認（入っていなければ安全に Wisdom）。これが「最後の安全網」。
CORSの壁：ブラウザから直接さくらを呼ぶと、安全上の仕組み（CORS）で弾かれることがある。 弾かれたら勝手に回避せず、まず相談するのが設計書のお約束。
（今回は弾かれず通った＝キーも有効、と確認できた 🎉）

4. AI②「翻訳＋やさしい一言」の中身

英語の名言を日本語に訳し、寄り添う一言を添える部分（translateQuote）。作りはAI①とほぼ同じ。違いは2つだけ。

違いAI①（タグ選択）AI②（翻訳＋一言）
渡す材料気持ち＋候補タグ名言・作者・気持ちの3つ
返ってくるものタグ1つ {"tag":…}2つ {"ja":…,"note":…}（訳とやさしい一言）

送る→待つ→受け取る→{ }を切り出す、という骨組みは全部同じ。一度わかれば使い回せる、という良い例です。

トーンの工夫：お願い文で「やさしく、いたわるように。説教くさい言い切りは避けて」と頼むことで、返ってくる一言の雰囲気を整えている（prd 7章）。

5. 覚えておきたい4つの考え方

言葉かみくだくと
APIキー（トークン）会員証。「私は使ってよい人」と示す合言葉。config.js に入れ、GitHubには上げない。
POST情報を相手に渡す種類の通信。「これ作って」と注文票を出すイメージ（取りに行くだけは GET）。
JSONデータを { "名前": 値 } で整理する書き方。AIへの注文も、AIの返事もこの形でやり取りする。
防御的パース相手が余計なものを付けても必要な部分だけ拾う守りの読み取り。{～} を切り出して読む。

6. 用語の言い換え表（出前のたとえ）

プログラムの言葉出前で言うと
URL注文する「お店の住所＋注文内容」のメモ
fetchその住所まで「頼んできて」という指示
await出前が届くまで「待つ」こと
response届いたけど、まだ袋を開けていない出前
.json()袋を開けて、食べられる（扱える）形にすること

関数 function材料を入れると結果を出す「機械」（たい焼き機）
id部品に付けた名札（あとで指で差せる目印）
jQuery の $('#…')名札を頼りに部品を「つかむ」道具

7. このアプリで完成したこと

  • 気持ちを入力 → 日本語の名言が表示される
  • 原文（英語）・作者・AIのやさしい一言も表示される
  • 気持ちを変えると、タグも名言も変わる
  • 失敗してもエラー画面が出る（白画面で固まらない）
  • さくらのAI Engine ＋ Quotable、2つのAPIが連携して動く
  • オープニング演出（タイトルがふわっと出て入力画面へ）
学びの核：「前のAPIの出力を、次のAPIの入力に渡す（＝チェーン）」を、自分の手で動かして体験できた。
